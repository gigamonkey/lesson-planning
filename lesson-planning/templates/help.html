How this works

This tool helps you take an official curriculum outline (an AP CED or the IB syllabus), draft your own raw objectives, map them onto that outline so you can see what's covered and what's missing, and organize them into a traceable lesson plan.

Load a reference→ Draft objectives→ Map coverage→ Build the plan→ Check gaps & trace

The pieces

Course
The top-level thing you're planning (CSA, CSP, IB). Each course has a reference hierarchy and, once you start planning, an outline.
Reference hierarchy
The official outline — an AP Course & Exam Description or the IB syllabus — loaded from a markdown file and treated as read-only. Its leaves (the deepest items, e.g. essential-knowledge statements) are the things you must cover.
Raw objective
An atom you draft — one concrete thing students should be able to do. Raw objectives live in a per-course pool until you place them.
Coverage
A link saying “this objective covers that reference leaf.” One objective can cover several leaves and a leaf can be covered by several objectives.
Outline (the plan)
Your authored lesson plan: units that contain lessons. Dropping a raw objective into a lesson is placement — and placement is just coverage into the plan, so the same objective ties your lesson back to the official leaf it covers.
Learning objective
The single whiteboard statement for a lesson, authored from the raw objectives you placed in it.

Reading a leaf's status

On the Outline page every reference leaf is badged with where it is in the lifecycle:

gap No objective covers this leaf yet — a hole in your coverage.
objective An objective covers it, but that objective isn't scheduled in a lesson yet.
planned An objective covers it and is placed in a lesson — fully planned.

The pages

Outline
The official outline as a collapsible tree, each leaf badged and showing the objectives mapped to it. Click an empty (red) leaf to add an objective right there. ?filter=gaps (the “Gaps only” toolbar link) prunes to just the uncovered leaves.
Objectives
The raw-objective pool. Edit objective text inline, and add or remove the reference nodes each one covers (the chips).
Plan
Build the lesson plan: create units, add lessons, and drag raw objectives from the pool into a lesson — or onto a unit's rough zone if you haven't picked the lesson yet. Give each lesson its learning objective.
Report
The traceability report: summary stats, the full gap list, and every covered leaf with its objective → lesson chain. Proof your plan covers the official outline.

Adding material in the app

  1. Draft an objective. On Outline, click a leaf's box and use its + to add an objective already mapped to that leaf; or add one on the Objectives page.
  2. Map coverage. On Objectives, add a node chip to map an objective to another leaf, or remove one to unmap it. On Outline you can drag an objective from one leaf to another to recategorize it.
  3. Build the plan. On Plan, “Add unit”, then + lesson, then drag raws from the pool into lessons. Write each lesson's learning objective in its italic field.
  4. Everything auto-saves to the live working database as you type or drag — there are no Save buttons.
Edits go to the live database immediately. To capture a reviewable, committable snapshot, click Export snapshot in the header — it writes the git-diffable .tsv files under lesson-planning/export/.

Loading a new reference or course

Reference hierarchies and courses come from markdown files, loaded from the command line (they're regenerated from those files, not authored in the app):

  1. Load the outline. uv run load_nodes.py csp/ced-hierarchy.md lesson-planning/db.db — it auto-detects the flavor, registers the course (e.g. csp) and its reference hierarchy (e.g. csp-ced), and loads the nodes. The course then appears in the top nav.
  2. Seed objectives (optional). uv run import_objectives.py <objectives.tsv> lesson-planning/db.db --course csp bulk-loads raw objectives and their coverage from a TSV.
  3. Then plan in the app as above.

Saving & version control

The database (db.db) is the live working copy and is not checked in. The committed state is the export directory of .tsv files. Export snapshot writes them; uv run rebuild_db.py reconstructs the database from scratch out of the schema, the hierarchy markdown, and that export — so the whole thing is reproducible from version-controlled inputs.
