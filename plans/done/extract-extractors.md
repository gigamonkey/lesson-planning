# Plan: Extract the hierarchy **extractors** into their own repo (with history)

## Goal

Pull the curriculum-hierarchy tooling out of `bhs-awesome` into a standalone git
repository that **preserves the development history** of every file it carries.

This is the **complement** of `plans/extract-lesson-planning.md`. That plan
carved out the *consumer* — the course-agnostic lesson-planning app — and
deliberately left behind the scrapers and all course data. This plan carves out
exactly what that one left behind: the code that **produces** the curriculum
hierarchies the app consumes — extracting them from the College Board CED PDFs,
the IB CS guide PDF, and the PreTeXt source books — together with those source
documents and the produced artifacts.

The two repos meet at one contract: the **numbered markdown hierarchy** format
(`# Unit N:` / `## 1.1` / `### 1.1.A` / …). This repo emits it; the
lesson-planning repo ingests it (`load_nodes.py`). `HIERARCHIES.md` already
documents this whole subsystem end to end and is, in effect, this repo's README.

## Scope (decided)

Unlike the lesson-planning extraction — which carried **no data** on purpose —
the extractors are *about* specific source documents, and their checked-in
outputs (the CED markdown especially) are the **source of truth**. So this repo
**does** carry data: the source PDFs and the hierarchy artifacts derived from
them. That's the point of the repo, not an accident.

Two decisions, both settled:

1. **Include the downstream builders** (not extractors-only). The strict reading
   of "extracting hierarchies *used by the app*" would stop at the markdown (the
   app reads markdown via `load_nodes`, not XML/DB). But `build_hierarchy_xml.py`
   / `build_hierarchy_db.py` + the `ced-to-html.xsl` / `Makefile` render path
   share `hierarchy.py` with the extractors, have **no other home** once
   lesson-planning is split off, and `HIERARCHIES.md` treats the whole thing as
   one subsystem. So the repo is "the curriculum-hierarchy toolkit"
   (source → markdown → XML/HTML/DB) — a self-contained unit. The keep-list
   reflects this.

2. **Carry the source data** — the PDFs and the produced `*-hierarchy.md` /
   `*.xml` / `ib-hours.tsv`. Without them the repo is scripts with nothing to run
   against, and the CED markdown is the canonical output that can't be
   regenerated deterministically (see the LLM workflows below). The PreTeXt book
   *trees* are the exception: they stay gitignored clones, regenerated by
   `just-pretext.sh`.

   **Caveat — redistribution.** Carrying the data is fine for **now** (a private,
   personal repo). It may not be fine to *publish*: the original College Board /
   IB PDFs are almost certainly copyrighted, and even the extracted hierarchies
   may be a derivative work whose redistribution isn't clearly permitted. So
   **before any public publish**, plan a second `filter-repo` pass over the new
   repo to strip the data from *all* history (not just the tip) — see "Before
   publishing publicly" below. Until then, keep everything.

## TL;DR — the approach

Same mechanism as the lesson-planning extraction: **work in a fresh local clone
and rewrite it with `git filter-repo`.** Don't rewrite the original repo; don't
hand-copy files (that loses history).

1. From a new directory, clone the source over the filesystem:
   `git clone /path/to/bhs-awesome hierarchy-extractors`.
2. Run `git filter-repo` with an explicit keep-list of the **toolkit** files plus
   the **source/artifact data** (and design plans). This drops everything else —
   the whole lesson-planning app, the quiz/deck/activity tooling, the objectives
   data — and prunes commits that no longer touch anything, keeping each
   surviving commit's original author, date, and message.
3. **Trim the few docs/configs that name the dropped pieces** (`pyproject.toml`
   deps, `HIERARCHIES.md`'s `load_objectives.py` + lesson-planning-schema
   sections, `.gitignore`, `CLAUDE.md`), confirm each extractor runs against its
   carried source, then point the clone at a new GitHub remote.

Why `filter-repo` (not `subtree split` / `filter-branch`): the keep-set is
scattered across the repo root, `csa/`, `csp/`, `ib/`, `workflows/`, and
`plans/`, so `subtree` (single-directory) can't express it; `filter-branch` is
deprecated and slow. `filter-repo` takes a path keep-list, unions it, and prunes
emptied commits. Install if missing: `pip install git-filter-repo` (or
`brew install git-filter-repo`). It refuses to run unless the repo is a fresh
clone — honor that; only ever rewrite the throwaway clone.

The shape is already flat (scripts at root, data under `csa/`/`csp/`/`ib/`), so —
unlike the lesson-planning extraction — there's **no Phase 2 reorganization**.
One pass.

`HIERARCHIES.md` is **kept and promoted** to the repo's primary doc (it already

```bash
extract_book_hierarchy.py
extract_book_text.py

# --- data: sources + produced artifacts (csa/ listed file-by-file) ---
csa/ced-2025.pdf

```

Notes for the executor:

- `csa/` is enumerated file-by-file (not as a whole dir) so `csa/mcqs.quiz` and
  `csa/learning-objectives/` are **excluded**. `csp/` and `ib/` are kept whole
  (every tracked file in them is hierarchy data). Verify the exact flag/format
  against the installed `filter-repo` version; equivalently use repeated
  `--path …` args.
- `--prune-empty always` drops commits that, after filtering, touch nothing
  (e.g. a commit that only edited the lesson-planning app or a quiz).
- `filter-repo` removes the `origin` remote on purpose — add a new one in step 6.
- Because excluded paths are stripped from *every* commit, no lesson-planning
  code, no objectives data, and no quiz/deck content survives anywhere in the new
  history — not just at the tip.

## `hierarchy.py`: ownership + the data-driven boundary

`hierarchy.py` (the markdown-hierarchy parser) is needed by both repos, but **not
symmetrically**, and we resolve the duplication by making the two libraries share
**data, not code**.

What each side actually uses:

- **This (extractor) repo** — only the two builders import it, and only
  `LEVEL_TAGS` + `parse_sections`. The `extract_*` scripts don't touch it.
- **The lesson-planning repo** — `load_nodes.py` uses `parse_sections`,
  `LEVEL_TAGS[flavor][level]` (in one spot, to turn a level *number* into the tag
  *string* it stores in `nodes.level`), and `hierarchy_title()` (a trivial
  `f"{course.upper()} {kind_label(...)}"` display helper). Its `FLAVOR_META`
  (flavor → course/kind/slug) is already local app policy, not format knowledge.

So the **only** flavor knowledge that crosses into the consumer's data is each
node's **level tag name**; everything else it needs is generic tree shape (id,
parent, ordinal, leaf-ness, text).

### Decision

1. **This repo owns `hierarchy.py`.** The format is *defined by what the
   extractors emit*, so the parser and flavor table (`LEVEL_TAGS`,
   detection regexes, `parse_sections`) live here. Trim the app-only
   `kind_label`/`hierarchy_title` out of this copy (step 4).

2. **The eventual cross-repo contract is data, not a shared module:** the
   extractor library emits the **already-parsed, already-tagged hierarchy as
   JSON**, and the lesson-planning library consumes it as a **dumb loader** that
   imports nothing from `hierarchy.py`. Shape:

   ```json
   { "flavor": "csa",
     "nodes": [
       {"id": "1",       "level": 1, "tag": "unit",                "parent": null,    "text": "..."},
       {"id": "1.1",     "level": 2, "tag": "topic",               "parent": "1",     "text": "..."},
       {"id": "1.1.A",   "level": 3, "tag": "learning-objective",  "parent": "1.1",   "text": "..."},
       {"id": "1.1.A.1", "level": 4, "tag": "essential-knowledge", "parent": "1.1.A", "text": "..."}
     ] }
   ```

   With the `tag` resolved per node, the consumer needs zero flavor knowledge and
   zero markdown parsing. The shared surface drops from "the same parsing module"
   to "a small, stable JSON schema," and drift becomes structurally impossible
   because one side generates it. (`parent`/`is_leaf`/`ordinal` can be either
   pre-computed by the producer or derived by the consumer — both are generic;
   pick when implementing.)

This was chosen over a shared *flavor-registry* JSON that both sides parse
against (datafies only `LEVEL_TAGS` + the detection regexes, but keeps a parser —
and an awkward regexes-as-data mini-DSL — on both sides, and still must be
vendored/synced; it relocates the duplication rather than removing it).

### Scope: now vs. follow-up

- **Now (this extraction):** own `hierarchy.py` here and trim the app-only
  helpers. Nothing else changes — the JSON emit is *not* required to ship this
  repo. The point is only to **not foreclose** the data-driven boundary.
- **Follow-up (separate, cross-repo):** add a `markdown → nodes JSON` emit step
  to this library (a small CLI/function over `parse_sections` + `LEVEL_TAGS`),
  rewrite `load_nodes.py` in the lesson-planning repo to load that JSON, and then
  **delete the lesson-planning repo's `hierarchy.py` copy entirely**. Track this
  in both repos' plans when scheduled.

**Tradeoff to remember:** today `load_nodes` reads hand-editable markdown
directly. Under the JSON contract, a hand-edit to a hierarchy must pass back
through this library's parse→emit step to refresh the JSON. That centralizes all
parsing in one place (good) at the cost of an extra emit step for hand-authored
hierarchies — acceptable, and the reason the emit step ships as an easy CLI.

## Before publishing publicly

Step 6 publishes a **private** repo, which carries the data. If this repo is ever
made public, the carried PDFs and extracted hierarchies must come out of the
**entire history** first (a `.gitignore` or a tip-only `git rm` is not enough —
the blobs stay reachable in old commits). Run a second `filter-repo` pass on a
fresh clone of the new repo:

```bash
git clone <private-repo> publish-prep && cd publish-prep
git filter-repo \
  --path csa/ced-2025.pdf --path csa/ced-2025-hierarchy.md --path csa/ced.xml \
  --path csa/bhsawesome-outline.md --path csp/ --path ib/ \
  --invert-paths --prune-empty always
```

This rewrites SHAs again (expected — independent history, same as the first
pass). Afterward the toolkit still runs, but against *your own* supplied
documents rather than the bundled originals — so add a short "bring your own
source PDFs / book tree" note to the README at that point, and decide separately
whether any hierarchy markdown is safe to keep as an example. **Settle the
licensing/redistribution question before this pass, not during it.**

## Decisions left to confirm

- **JSON node-list schema details** — `parent`/`is_leaf`/`ordinal` pre-computed

## Loose ends to handle during execution

- **`hierarchy.py` ownership** — resolved in its own section above (this repo
  owns it; the cross-repo boundary becomes emitted JSON, not shared code). Until
  the follow-up lands, the lesson-planning repo's copy keeps working unchanged;
  note the ownership + the planned JSON contract in this repo's
  `CLAUDE.md`/`HIERARCHIES.md`.
- **The CED markdown is canonical and not deterministically reproducible.** The
  `workflows/*.js` LLM pipelines regenerate `csa/ced-2025-hierarchy.md` /
  `csp/ced-hierarchy.md`, but re-running can produce small wording differences,
  so the checked-in markdown is the source of truth (the smoke test diffs only
  the *deterministic* extractors' outputs, not the CED workflows').
- **Large PDFs in history.** The carried PDFs (≈4.7/7.5/3 MB) are already in the
  source history, so keeping them adds no new bloat beyond what's there. They're
  the inputs the extractors exist to read — expected.
- **After extraction, decide the fate of this tooling in the old repo** (leave it
  in place, or remove it and link to the new repo). This interacts with the
  lesson-planning split — coordinate the two. Separate cleanup, out of scope here.
